| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 文書の読み手（属性や特性）を記述している | 対象文書の冒頭において、想定する読者の職種、役割、および前提となる業務スキルや専門知識（プログラミング言語、ドメイン知識など）を具体的に定義している。 | ★新規追加【改行】【非適合例】本仕様書は、システム開発に関わる全担当者を対象とする。（読み手の役割や必要とする経験値、スキルに関する記述が曖昧であり、不十分である。）【改行】→【適合例】本仕様書は、基本設計工程を担当するアプリケーションエンジニアを対象とする。JavaおよびSpring Bootに関する基礎知識を有していることを前提とする。 |
| 文書の読み手の範囲（提出先,配布先,開示範囲）を記述している | 対象文書の管理情報として、作成した文書や成果物に対する社内外の提出先、関係者への配布先、および機密区分に応じた情報の開示範囲を明記している。 | ★新規追加【改行】【非適合例】プロジェクト計画書において、各開発フェーズの具体的な成果物やスケジュールが記載されているが、その取り扱い方針が指定されていない。（成果物に対する提出先などの、読み手の範囲に関する記述が欠落している。）【改行】→【適合例】プロジェクト計画書において、各開発フェーズの成果物ごとに、提出先（顧客情報システム部）、配布先（プロジェクト内全メンバー）、および開示範囲（社外秘）を一覧表として定義している。 |
| 文書の適用範囲を記述している | 文書の前提条件として、システム化の対象となる業務領域および機能範囲と、開発対象外となる境界線（アウトオブスコープ）を明確に定義している。 | ★新規追加【改行】【非適合例】システム要件定義書において、顧客からヒアリングした業務要望を無差別に要件として列挙している。（適用範囲が不明確であるため、システムの目的と無関係な要件が多く含まれる原因となっている。）【改行】→【適合例】システム要件定義書において、国内向け販売管理業務における受注から出荷までの領域を適用範囲として明記し、境界線外の業務は対象外としている。 |
| 文書の目的を記述している | 当該文書を作成する背景、および読者がその文書を用いて達成すべき目的や得られる結果を、文書の冒頭に明記している。 | ★新規追加【改行】【非適合例】ソフトウェア設計書の冒頭から、画面レイアウトやデータベースの仕様が直接羅列されている。（文書自体の目的が記載されていないため、目的と無関係な機能や仕様が混入する原因となっている。）【改行】→【適合例】ソフトウェア設計書の冒頭において、「本設計書は、システムの要件を満たすアーキテクチャの設計方針を定義し、開発チーム全体で共有することを目的とする」と明記している。 |
| 文書の目的を達成するために必要な項目を記述している | 要求定義を行う文書において、ステークホルダー間でシステムの方向性を合意するため、背景となる開発の目的と対象スコープを必須項目として明記している。 | ★新規追加【改行】【非適合例】要求仕様書の冒頭において、次期システムの機能一覧表のみが記載されている。（システムの方向性を決定づける開発の目的が明示されていない。）【改行】→【適合例】要求仕様書の「はじめに」の章において、業務効率化という開発の目的と、本プロジェクトにおけるシステム化のスコープを明確に記載している。 |
| 文書の目的を達成するために必要な項目を記述している | 要求定義書において、システムの品質基準を総合的に担保するため、機能要件だけでなく非機能要求（性能、セキュリティなど）を網羅的に記載する専用の章を設けている。 | ★新規追加【改行】【非適合例】要求仕様書において、ユーザーが利用する画面や帳票に関する機能要件のみが記載されている。（品質担保の目的達成に必要な、非機能要求に関する項目が欠落している。）【改行】→【適合例】要求仕様書において、機能要件の章とは別に非機能要求の専用章を設け、性能要件やセキュリティ要件を網羅的に記載している。 |
| 文書の目的を達成するために必要な項目を記述している | 設計関連文書において、各開発フェーズの役割を果たすため、当該工程で達成すべき技術的課題の解決方針や業務的な設計の目的を、必須項目として明記している。 | ★新規追加【改行】【非適合例】基本設計書の冒頭から、システム構成図やインターフェースの定義が直ちに記載されている。（当該フェーズにおける設計の目的や方針が欠落している。）【改行】→【適合例】基本設計書において、「本設計は、将来的なデータ量増加に耐えうる拡張性の高いデータベース構造を定義することを目的とする」と、設計の技術的な目的を記載している。 |
| 文書の目的を達成するために必要な項目を記述している | 設計関連文書において、システムの保守性および拡張性を担保するため、初期開発時の要件だけでなく、将来的な改訂や機能追加を見据えた設計方針を必須項目として記述している。 | ★新規追加【改行】【非適合例】現在の業務要件を満たすための、短期的な実装方針のみを記載している。（システムの保守性や、将来の改訂に関する観点が欠落している。）【改行】→【適合例】基本設計書において、将来の法改正に伴う帳票レイアウトの改訂を想定し、表示ロジックとデータ出力処理を分離する方針を記述している。 |
| 文書の目的を達成するために必要な項目を記述している | 議事録において、会議の記録と合意形成という文書の目的を達成するため、開催日時、参加者、議題、および決定事項を漏れなく記載するための定型フォーマットを設けている。 | ★新規追加【改行】【非適合例】議事録のテンプレートに、自由記述のメモ欄のみが存在する。（合意形成という目的を達成するために必要な、決定事項などを記載する専用欄がない。）【改行】→【適合例】議事録のフォーマットにおいて、ヘッダ部分に開催日時、参加者、議題を記載する欄を設け、末尾に決定事項および保留事項を記載する欄を定義している。 |
