### システム開発文書品質モデル：再構成指示プロンプト（260410a・正統進化版 for Gemini 3.1 Pro）

--------------------------------------------------------------------------------

#### 【0. タスク定義】
本タスクは **「構造変換処理」** である。創作・要約・解釈ではなく、以下の原則で実行すること。
* 判断に迷った場合は **元データ優先** で解釈する
* 欠落を補う場合は **最小限の補完** に留める
* ただし、ルール A（メタ記述の具体化）・ルール F（パラグラフ構造の例示）、および **「測定項目」と元の内容にズレがある場合の「測定基準」「記述例」の改訂・新規作成** においては、その限りで具体的な実務文を創作してよい。
**注記** ：「創造性禁止」は、AIが恣意的に内容を変えることへの禁止であり、ルールで明示的に許可された補完・創作・補正と矛盾しない。ルールに基づく補完はこの制約の対象外である。

--------------------------------------------------------------------------------

#### 【0.5 深層推論フェーズ（出力前の内省プロセス）】
テーブル出力を生成する **前に**、以下の3点を脳内で推論し、その結果を「測定基準」と「記述例」の品質に反映させること。
1. **リスク分析**: この「測定項目」が遵守されない場合、システム開発の現場（設計・実装・テスト・保守）で具体的にどのような手戻り・バグ・品質劣化が生じるか？
2. **価値定義**: この項目を「具体化・尺度化」することで、文書の読み手（レビュアー・後工程の開発者）のどのような判断が正確になるか？
3. **現場具体化**: 元データが抽象的な場合、現在のエンタープライズ開発（API連携、クラウドネイティブ、マイクロサービス、非同期処理、CI/CD、AI統合等）の文脈にどう当てはめると最も説得力があるか？

これらの思考を経て、単なる「言葉の変換」ではない「生きた知見」を反映させること。

--------------------------------------------------------------------------------

#### 【1. 最優先・絶対遵守ルール】

##### ルール 1：測定項目の完全維持
「測定項目」列の文字列は、提供データの通り、一字一句変えずにそのまま転記してください。

##### ルール 2：空白セルの文脈補完
元のテーブルで「品質副特性」「測定項目」の列が空白の場合は、直近の上の行の値をそのまま引き継いで補完してください。（推測・補間は禁止。機械的にコピーするだけです。）

##### ルール 3：既存データの原則保持と例外（意味のズレの補正）
元の「例」「違反例」に含まれる具体的な文言・数値・例文番号は、原則として削除せず「記述例」のベースとして活用します。
**【重要・例外処理】** ：ただし、元の「例」「違反例」の内容が「測定項目」の意図に対して不十分である、または意味がズレている場合は、元の内容の維持よりも **「測定項目」との整合性を優先** してください。その際は「測定基準」を測定項目に合わせて改定し、それに基づいた「記述例」を新規作成してください。

##### ルール 4：Markdown テーブル出力（Excel 置換前提）
* 出力形式は **Markdown テーブル** とします。表内に情報元の引用番号などは絶対に出力しないでください。
* セル内での物理的な改行（Enter）および `<br>` タグは **絶対に禁止** します。
* 改行が必要な箇所には、必ず **`【改行】`** という文字列を挿入してください。
* **【重要】** `【改行】` の前後にスペース（空白）は絶対に入れないでください。Excelでの置換時に不要な空行が発生する原因となります。

##### ルール 5：アトミック分割（行分割）の徹底
以下のいずれかに該当する場合は、 **必ず行を分けて** 最小単位で出力してください。
| 条件 | 処理 |
| ------ | ------ |
| 測定基準に複数の判定ポイント（A かつ B）がある | A と B でそれぞれ独立した行に分割する |
| 1 つの測定項目に複数の独立した違反類型がある | 違反類型ごとに行を分割する |
| 独立した例文セット（例文1a と 1b など）が混在している | 例文ペアごとに行を分割する |

#### 【2. 出力列の定義】

##### 列 1：測定項目
元のデータのまま転記します。

##### 列 2：測定基準
測定基準とは、測定項目を客観的に判定するための「合否判定の尺度（ものさし）」です。ここが不鮮明だと品質が測定できません。以下の手順で生成してください。

**【Step 1】元データと測定項目の整合性を評価する**
元データ（例・違反例）が、「測定項目」の意図を十分に反映しているかを厳格に評価します。
* 反映している場合 → **Step 2** へ進む
* 反映していない（不十分・ズレている）場合 → **Step 3** へ進む

**【Step 2】元データから高品質な測定基準を抽出する（整合している場合）**

1. 元の「例」と「違反例」から、判定の根拠となる「コアとなる技術的ルールや観点」を抽出する。
2. 抽出した観点を、判定の尺度として言語化する。具体的すぎる固有名詞（文書名等）は汎用語に置き換える。

**【Step 3】判定基準に「新しい情報」を付加し昇華させる（絶対必須）**
測定項目（A列）を単に言い換えただけの文は「失敗」とみなします。AI自身のソフトウェア工学・文書品質の専門知識を動員し、判定を客観的に行うための**「プラスアルファの情報」**を必ず追加してください。

**【Step 3 補足：汎用ワード（逃げ言葉）の単独使用禁止】**
以下の言葉を、具体的な条件や指標を伴わずに単独で使用することを禁じます。
* NG例: 「適切に」「正しく」「十分に」「適宜」「必要に応じて」「適切と思われる」
* OK例: 「上位設計方針に基づき」「外部APIのレスポンス定義と矛盾なく」「JTF日本語標準スタイルガイドに準拠して」
具体的な数値、規格名、ドメイン知識（設計原則、テスト基準、セキュリティ標準等）に基づいた尺度を導入すること。

*   **具体化の観点例**:
    *   **対象範囲**: 「全章節、図表、専門語彙、パラメータ設定、物理構成図、APIシーケンス図」など、具体的に何を確認すべきか。
    *   **判定の拠り所**: 「上位設計方針、用語定義集、外部インタフェース仕様、業務フロー、セキュリティ基準、JTFスタイルガイド」など、何と比較すべきか。
    *   **合格の指標**: 「一貫しているか、網羅しているか、最新の状態か、根拠が明記されているか、実装コードと1対1で対応しているか」など、どのような状態が合格か。
これらを組み合わせ、A列よりも「情報密度の高い」実務的な基準を独自に定義してください。

**【Step 4】測定基準の記述制約（絶対遵守）**
* **客観的尺度化**：単なる状態説明（〜と書かれている等）ではなく、合否を客観的に判定できる基準（〜を明記している、〜と定義している等）として記述すること。
* **読みやすさと句読点**：技術文書として適切な位置に**読点（、）**を入れ、文末には必ず**句点（。）**を付けること。
*   **下位要素として具体化・詳細化**：「測定基準」は「測定項目」を判定するための詳細なルールです。語彙の置換（単なる言い換え）を厳禁し、A列には含まれていない「判定に役立つ具体的な観点」を少なくとも1つは含めること。
* **1文1動作**：測定基準は「A かつ B」を避け、一項目一動作に分解すること。
* **語尾表現**：状態を判定するため、「～記述している」や「～を用いている」など、能動かつ状態を示す語尾で統一すること。

##### 列 3：記述例（適合例／非適合例）
元の「例」「違反例」を以下の形式で整理し、**「代表的で理解しやすい」代表例**として提示します。
`【非適合例】(文） (非適合の理由）【改行】→【適合例】(文）`

* **具体例の保持**：列2で「測定基準」から除外した具体的な対象（文書名、処理名、固有の条件など）を、この「記述例」の列に必ず再注入・保持すること。
* **転記原則**：元の「例」「違反例」の文言はそのまま転記することを原則とします。ただし、メタ記述（状態説明）のみの場合は具体文に変換し（ルール A）、Step 3を適用した場合は新しい基準に合致する記述例を新規作成します。
* 新規作成・補完した場合は文頭に `★新規追加【改行】` を付与します。

--------------------------------------------------------------------------------

#### 【3. 記述例の作成ルール（A〜F）】

##### ルール A：メタ記述の「具体化」（補完が許可される場面）
元のデータが「〜が記載されていない」などの **状態説明のみ** の場合は、システム開発の実務（要件定義書・基本設計書・議事録など）に即した **具体的なドキュメントの一文** を具体例文として創作してください。

##### ルール B：修正指示コメントの除去
元の違反例に「（〜が曖昧）→『例』列の [例文 1a] に修正」などの記述がある場合、「〜が曖昧」という **理由部分のみ** を抽出して `(非適合の理由)` とし、「→ 〜に修正」という作業指示は出力から完全に除外します。

##### ルール C：「★新規追加」ラベルの付与
元のデータに適合例・非適合例の片方しかない場合や、メタ記述の具体化（ルール A）や基準改定（Step 3）に伴い例文を新規創作した場合は、 **記述例セルの文頭にのみ** `★新規追加【改行】` を付けます。

##### ルール D：不完全な文の補強
元の例が短すぎて文脈が不明な場合は、元の語彙を維持しつつ、システム開発ドメインとして意味が通る実務的な一文に補強します。

##### ルール E：列挙型データ・コード例の処理
プログラムのコード例や事実の箇条書きなどが含まれる場合は、無理に1行にまとめず、`【改行】` を用いて元の構造を維持したまま提示します。

##### ルール F：パラグラフ構造の例示
「主題文→支持文の構成になっている」などの抽象的な構造指示がある場合は、実際のシステム仕様を題材にした具体例文セットを創作します。

#### 【4. 文章品質の制約】
| 制約項目 | 内容 |
| ------ | ------ |
| 基準と具体例の分離 | 測定基準には、E-mailや特定の機能名などの具体的すぎる対象物を含めず、汎用的なルールとして記述すること。具体的な条件や固有の要素は「記述例」の側に保持・反映させること。 |
| 冗長なマージ禁止 | 「例」と「違反例」が単なる裏表の意味である場合、「〜を避けて、〜している」と両方を並べた冗長な文にせず、簡潔な肯定形の基準にすること。 |
| 能動態の徹底 | 適合例では受身形を避け、可能な限り能動態（〜する）で記述すること。 |
| 一文一義 | 記述例自体が品質違反（長文・複雑な係り受け）を起こさないよう、簡潔な文にすること。 |
| 逃げ言葉の排除 | 測定基準において「適切に」「正しく」「十分に」等の汎用ワードを具体的条件なしに単独使用しないこと。（Step 3 補足を参照） |

--------------------------------------------------------------------------------

#### 【5. 特殊パターン別の処理方針】
* **パターン 1（複数類型）** ：「複数の解釈が成り立たない」など、1つの測定項目に複数の違反類型がある場合は、類型ごとに **行を分けて** 出力します。
* **パターン 2（修正対応）** ：「[例文 1] … → [例文 1a] に修正」は、例文1を非適合例、例文1aを適合例として出力し、修正指示文字は除外します。
* **パターン 3（コード複数行）** ：「記述に重複がない」のコードブロックなどは `【改行】` で構造を維持し、非適合例側には簡潔なメタ記述例を創作します。
* **パターン 4（列挙形式）** ：事実・意見などの箇条書きは `【改行】` で区切りながら整形し、省略せずすべて含めます。
* **パターン 5（片方欠落）** ：適合例のみ・違反例のみの場合は欠落側を創作し、`★新規追加【改行】` を付けます。

#### 【6. 出力前の自己検証チェックリスト】
出力する前に、以下の項目をすべて確認してください。1 つでも違反があれば出力を修正してください。
| # | 確認項目 |
| ------ | ------ |
| 1 | 測定項目が元データと完全一致しているか |
| 2 | 測定基準が測定項目の「下位要素」として具体化されており、客観的な「判定の尺度」になっているか |
| 3 | **測定基準および記述例において、読点（、）と句点（。）が適切に使用されているか** |
| 4 | **測定基準に「E-mail」や「特定の機能名」など、具体的すぎる要素が混入していないか（具体例は記述例の列に分離されているか）** |
| 5 | セル内に `【改行】` 以外の物理的な改行が存在しないか。また `【改行】` の前後に不要なスペースがないか |
| 6 | メタ記述が具体的な実務文に変換されているか（ルール A） |
| 7 | アトミック分割の条件を満たす行が、正しく分割されているか（ルール 5） |
| 8 | **測定基準に「適切に」「正しく」「十分に」等の汎用ワードが、具体的条件なしに単独使用されていないか** |
