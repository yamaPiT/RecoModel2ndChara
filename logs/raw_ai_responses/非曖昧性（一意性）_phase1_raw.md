| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 複数の解釈が成り立たない | 順次処理と並行処理のどちらを意図しているかを明確にするため、文中に曖昧な接続助詞を含めず、実行順序または論理関係を明示している。 | ★新規追加【改行】【非適合例】APIからデータを取得し、DBを更新する。(「〜し／して」などの中止法により、順次実行か並行実行か、あるいは原因と結果かが不明である)【改行】→【適合例】APIからデータを取得した後、そのデータをDBに反映する。 |
| 複数の解釈が成り立たない | 条件分岐の判定において、全体否定と部分否定の違いによる実装時の条件漏れを防ぐため、論理演算子（AND、OR）の意図が一意に特定できる表現を用いている。 | 【非適合例】[例文2]全ての処理が成功しない場合は手順Cに進む。(全体否定か部分否定かがわからない)【改行】→【適合例1】[例文2a]全ての処理が失敗した場合は手順Cに進む。【改行】→【適合例2】[例文2b]どれか1つの処理が失敗した場合は手順Cに進む。 |
| 複数の解釈が成り立たない | 仕様の前提か決定事項かを明確にするため、既存の仕様を参照しているのか、当該ドキュメントで新たに仕様を定義しているのかを区別して記述している。 | 【非適合例】[例文1]プログラムの構造を図1に示す。(すでに決まっているプログラムの構造を図1に記載しているのか、図1を使ってプログラムの構造を決定しているのかが曖昧である)【改行】→【適合例1】[例文1a]以上に述べたプログラム構造を図示すると図1となる。【改行】→【適合例2】[例文1b]図1によりプログラムの構造を定める。 |
| 複数の解釈が成り立たない | 複数の修飾対象が存在する文において、誤った対象への修飾による仕様の誤解を防ぐため、修飾語と被修飾語の対応関係を読点や語順により一意に特定できる構造としている。 | ★新規追加【改行】【非適合例】暗号化されたDBのパスワードとユーザーIDを取得する。(係り受けの関係において、暗号化されている対象がパスワードだけか、ユーザーIDも含むのかが明確でない)【改行】→【適合例】DBのパスワード（暗号化済）と、ユーザーIDを取得する。 |
| 複数の解釈が成り立たない | 複雑な文構造による可読性の低下を防ぐため、主語と述語、および修飾語と被修飾語を近接して配置し、間の挿入句を独立した別の文に分離して記述している。 | ★新規追加【改行】【非適合例】システムは、外部APIから最新のユーザー情報を取得し、取得データ内でバリデーションエラーが発生しなかった場合のみ、DBの該当レコードを更新する。(主語と述語の距離が離れすぎている)【改行】→【適合例】システムは、外部APIから最新のユーザー情報を取得する。取得データ内でバリデーションエラーが発生しなかった場合のみ、システムはDBの該当レコードを更新する。 |
| １つの文内で１つの事柄のみを記述している | エラーハンドリングや正常系の処理手順が実装時に漏れることを防ぐため、一つの文に複数の条件や動作を連結せず、単一の操作や状態遷移ごとに分割して記述している。 | ★新規追加【改行】【非適合例】ユーザーがログインボタンを押下した際、入力値の検証を行い、エラーがあれば画面にメッセージを表示して、正常であればダッシュボード画面に遷移し、最終ログイン日時を更新する。(1つの文の中に複数の事柄が混在している)【改行】→【適合例】ユーザーがログインボタンを押下した際、システムは入力値の検証を行う。エラーがある場合は、画面にメッセージを表示する。正常な場合は、ダッシュボード画面に遷移する。遷移後、最終ログイン日時を更新する。 |
| 同じ内容のものを違う名前で定義しない。 | ドメインモデルと実装コード間の不一致を防ぐため、プロジェクトの用語定義集やデータディクショナリに従い、同一の概念・データ項目・画面コンポーネントには文書全体で同一の名称を用いている。 | ★新規追加【改行】【非適合例】顧客IDを入力欄に設定する。その後、ユーザーIDを利用して外部APIを呼び出す。(同じ概念に対して異なる名称が使われている)【改行】→【適合例】顧客IDを入力欄に設定する。その後、顧客IDを利用して外部APIを呼び出す。 |
| 指し示すものが明確である | 処理の再利用や参照関係における特定漏れを防ぐため、「これ」「前回」などの抽象的な指示語を避け、具体的な機能名やトリガー条件を明記している。 | 【非適合例】[例文1]前回の処理と同様とする。(指示語「前回」が何を指すのかを明示していない)【改行】→【適合例】[例文1a]承認ボタンが押下されたときと同じ処理をする。 |
| 指し示すものが明確である | ドキュメントの参照時期によって生じる解釈のズレを排除するため、日付や期間を「昨年」「来月」などの相対表現ではなく、西暦を用いた絶対日付で明記している。 | 【非適合例】[例文2]昨年12月のリリース記録を参照する。(直示表現「昨年」が文書の閲覧時期により曖昧になる)【改行】→【適合例】[例文2a]2023年12月のリリース記録を参照する。 |
| 表に空欄を残さない | 設定値やパラメータの定義漏れによる実装待ちを防ぐため、値が存在しないテーブルのセルには空白を残さず、「該当なし」や「N/A」などの明示的な文字列または記号を記述している。 | ★新規追加【改行】【非適合例】データ定義書の「初期値」列において、何も入力されていない空白のセルが存在する。(表内の空欄が、該当なし、TBD、記入忘れのいずれの意味にも解釈できる)【改行】→【適合例】データ定義書の「初期値」列において、初期値を持たない項目のセルには「N/A」または「-（ハイフン）」を記入し、値がないことを明示している。 |
| 状態とイベント（事象）を区別している | 状態検知（レベルトリガー）と状態変化の検知（エッジトリガー）の混同による誤動作を防ぐため、継続している「状態」と、瞬間的な「状態の遷移（イベント）」を明確に区別する用語を用いている。 | 【非適合例】[例文1]電源ONのときに、バッチ処理を実行する。(「ONの状態のとき」か、「ONに変わったとき」というイベントの表現かが曖昧である)【改行】→【適合例】[例文1a]電源がOFFからONに変わったときに、バッチ処理を実行する。 |
