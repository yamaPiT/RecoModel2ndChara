| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 原因と結果が対応している | 特定の事象が発生した際のシステムの動作を、入力（原因）と出力（結果）の因果関係が明確になるよう一貫して記述している。 | 【非適合例】認証エラー発生時、ログイン画面に戻る場合とエラー画面を表示する場合がある（箇所により対応が不整合）。【改行】→【適合例】認証エラー発生時は、一貫してエラーコード「E001」をダイアログ表示し、ログイン画面へ遷移する。 |
| 理由または根拠を記述している | 特定の設計判断や技術選定を行う際、その決定に至った背景、制約、または比較検討結果などの客観的な根拠を明記している。 | ★新規追加【改行】【非適合例】DBサーバーにはPostgreSQLを採用する（選定理由が不明）。【改行】→【適合例】同時実行制御の要件を満たし、かつライセンスコストを抑制するため、オープンソースのPostgreSQLをDBサーバーに採用する。 |
| 記述間の接続関係（展開、順序）が適切である | 処理手続きや一連のイベントを、業務の発生順序やプログラムの実行時間軸などの論理的な時系列に沿って記述している。 | 【非適合例】データ更新後にログイン確認を行い、その前に入力内容のチェックを行う（処理手順が実行順序と一致していない）。【改行】→【適合例】1.入力内容をチェックする。2.ユーザーの権限を確認する。3.DBのデータを更新する。 |
| 全体像を明示した後に詳細を記述している | 章や節の冒頭において、その項で説明する対象の全体像や結論を提示してから、具体的な詳細説明や根拠を記述する構成をとっている。 | 【非適合例】（3ページにわたる技術解説の末尾に）以上より、本機能のAPI実装は不要と判断する（結論が後回しで意図が不明）。【改行】→【適合例】本章では、既存APIの再利用性について検討した結果、新規開発を不要とする判断に至った経緯を述べる。詳細は以下の通りである。 |
| 全体像を明示した後に詳細を記述している | パラグラフ（段落）ごとに、核心となる要点を述べた主題文を先頭に配置し、それに続く支持文で補足説明を行う構成を維持している。 | ★新規追加【改行】【非適合例】バックアップは毎日2時に行う。容量は10GBである。リストアテストも月次で実施する（単なる事実の羅列で要点が不明）。【改行】→【適合例】データ保守運用は、可用性と整合性を重視して設計する。具体的には、毎日午前2時のフルバックアップに加え、月次でのリストア検証を必須とする。 |
| 記述単位の見出しと本文が合致している | 見出し（章・節・項）が示す内容と、その配下に記述されている本文の情報が正確に対応しており、情報の乖離がない。 | 【非適合例】見出し：エラー処理方針【改行】本文：正常系の画面遷移図を以下に示す（見出しの内容と本文が不一致）。【改行】→【適合例】見出し：エラー処理方針【改行】本文：例外発生時のロールバック手順およびユーザー通知の定義を以下に記述する。 |
| 図表と本文とに齟齬がない | 本文中で説明している名称、数値、ステータス遷移等の情報が、参照先の図表内の記載と矛盾なく一致している。 | 【非適合例】本文では「ステータスを保留にする」とあるが、状態遷移図内では「一時停止」と記載されている（用語の不一致）。【改行】→【適合例】本文で解説する「承認待ち」状態への遷移条件は、別紙のワークフロー図に記載された「承認待ち」の定義と一致している。 |
