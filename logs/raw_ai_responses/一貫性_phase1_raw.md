| 原因と結果が対応している | 特定の事象に対するシステムの挙動が、文書内のすべての記載箇所において同一の仕様として定義されている。 | ★新規追加【改行】【非適合例】第3章では「API通信エラー時はリトライ処理を3回実行する」と記述しているが、第5章のエラー定義表では「即時エラー画面へ遷移する」と記述している。（エラー処理について、異なる箇所で異なる対応が記述されている）【改行】→【適合例】API通信エラー時のシステムの応答について、全章を通して「最大3回のリトライ処理を行い、失敗した場合はエラー画面へ遷移する」と一貫した仕様を記述している。 |
| 理由または根拠を記述している | システム仕様の決定事項に対して、その方式を採用した理由や評価の根拠を明記している。 | ★新規追加【改行】【非適合例】ユーザー認証機能には、OAuth2.0による外部プロバイダ認証を採用する。（採用理由の記述がない）【改行】→【適合例】ユーザー認証機能には、既存の社内システムとの統合によるシングルサインオンを実現するため、OAuth2.0による外部プロバイダ認証を採用する。 |
| 記述間の接続関係（展開、順序）が適切である | システムの処理手続きやライフサイクルの説明を、実際の実行順序や時間の経過に沿って時系列で記述している。 | ★新規追加【改行】【非適合例】会員登録処理では、完了メールを送信する。その前にDBへユーザー情報をInsertするが、パスワードはハッシュ化しておく。（処理手順が順不同で記述されており、時間の流れに沿っていない）【改行】→【適合例】会員登録処理では、まずパスワードをハッシュ化し、次にDBへユーザー情報をInsertする。登録成功後に、完了メールを送信する。 |
| 全体像を明示した後に詳細を記述している | 文書の冒頭や各章の始めに、目次、アーキテクチャ図、または業務フローなどの全体構成を提示した上で、個別の詳細仕様を記述している。 | ★新規追加【改行】【非適合例】(A4で3ページにわたる個別のエラー条件の記述の後に) だから、この要求は却下する。(全体構成が提示されず詳細が先行している)【改行】→【適合例】冒頭で「本章では要求の却下理由について述べる」と全体像を明示した後に、A4で3ページにわたる詳細な分析や理由を記述している。 |
| 全体像を明示した後に詳細を記述している | 各段落において、最初の文で結論や主題を明示し、続く文でその理由や具体例を補足する構成で記述している。 | ★新規追加【改行】【非適合例】当システムは夜間バッチで大量データを処理する。既存のRDBではIOボトルネックが発生する。そのため、新たにインメモリDBを導入してパフォーマンスを向上させる。（主題である結論が最後にきている）【改行】→【適合例】当システムはパフォーマンス向上のため、新たにインメモリDBを導入する。夜間バッチにおける大量データ処理時のIOボトルネックを解消することが目的である。 |
| 記述単位の見出しと本文が合致している | 各章や節の見出しの名称が、その配下に記述された仕様やプロセスの内容と論理的に合致している。 | ★新規追加【改行】【非適合例】見出し「3.1 ログイン処理の正常系フロー」の配下に、パスワード間違い時のアカウントロック機能に関する仕様を記述している。（見出しが示す内容と本文の内容が異なっている）【改行】→【適合例】見出し「3.1 ログイン処理の正常系フロー」の配下には、正しいIDとパスワードの入力からダッシュボード画面遷移までの手順のみを記述している。 |
| 図表と本文とに齟齬がない | 図表内に記載された要素と、本文中の説明において、名称や処理内容の定義が完全に一致している。 | ★新規追加【改行】【非適合例】本文では「注文データは直接決済システムへ送信される」と記述しているが、業務フロー図では「注文データが一旦キューを経由して送信される」構成として描画されている。（図表に記載された内容と本文に記述された内容が矛盾している）【改行】→【適合例】業務フロー図で「注文データが一旦キューを経由して送信される」構成を描画し、本文でも図表と一致する内容で「注文データは直接送信せず、非同期処理のためのキューを経由する」と記述している。 |
