| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 関連する複数箇所の記述に対して、参照情報を明示している | 外部文書や他セクションの内容を引用または参照する場合、引用元の文書名、章節番号、またはリポジトリ上の格納位置を具体的に明記している。 | ★新規追加【改行】【非適合例】先行研究の結果に基づき、本システムのアーキテクチャを決定した。（引用元が明示されておらず、文書内の関係性が不明確）【改行】→【適合例】先行研究（[引用番号]：2023年度XX技術報告書）の結果に基づき、本システムのアーキテクチャを決定した。 |
| 関連する複数箇所の記述に対して、参照情報を明示している | 文書の改訂において、前バージョンまたは関連する他バージョンとの具体的な相違点や変更箇所を、履歴情報や比較表を用いて追跡可能な状態で記述している。 | ★新規追加【改行】【非適合例】最新の仕様に合わせて本章の内容を全面的に更新した。（具体的な変更点や旧版との差異が不明）【改行】→【適合例】前バージョン（v1.2）からの変更点として、認証エラー時のリトライ回数を3回から5回に変更した。 |
| 関連する複数箇所の記述に対して、参照情報を明示している | 文書内の修正箇所を一意に特定するために、対象となる文書名、版番号、作成日、および作成者をメタデータまたは修正履歴として記述している。 | ★新規追加【改行】【非適合例】不具合修正のため、基本設計書の一部を修正した。（修正対象の版や作成者が特定できない）【改行】→【適合例】基本設計書（文書番号：BD-001、版：v2.0、作成日：2024/03/25、作成者：山田）の第3章を、不具合報告書#123に基づき修正した。 |
| 関連する複数箇所の記述に対して、参照情報を明示している | 特定の機能や仕様の根拠を理解するために他文書の確認が必要な場合、参照すべき具体的な文書名や管理番号を明記している。 | ★新規追加【改行】【非適合例】基本設計書の機能F1の必要性を理解するには、要求仕様書を確認する必要がある。（具体的な要求番号が不明）【改行】→【適合例】基本設計書の機能F1の必要性を理解するには、関連する要求仕様書のR1、R2を読む必要がある。 |
| 関連する複数箇所の記述に対して、参照情報を明示している | 文書内で散在している複数の情報を要約・集約して提示する場合、元の記述箇所（表番号、図番号、節番号等）との対応関係を明示している。 | ★新規追加【改行】【非適合例】各センサの入力周期は、前述した通りである。（どの記述を指しているか不明）【改行】→【適合例】センサDを10msec周期で入力する。以上に述べたセンサA、B、C、Dの入力周期を表3にまとめる。 |
| 記述内容は追跡可能である | 上流工程の要件（要求ID等）と下流工程の設計要素（機能ID等）の対応関係を、トレーサビリティマトリクスやIDの埋め込みによって、双方向に特定できる状態で記述している。 | ★新規追加【改行】【非適合例】基本設計書と詳細設計書の間で機能の対応関係が管理されていない。（設計変更時の影響範囲の追跡が困難）【改行】→【適合例】要求仕様書の要求R1を修正する際に、対応する基本設計書の機能F1と詳細設計書の機能詳細FS1を即座に特定できるよう、トレーサビリティマトリクスで紐付けを管理している。 |
