| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 箇条書きのルールに適合している | 文書内で定義されたスタイルガイドやプロジェクト規定に基づき、各階層の箇条書き記号（中黒、番号、記号等）が、文書全体で統一して適用されている。 | ★新規追加【改行】【非適合例】第1章では「・」を使用し、第2章では「-」を箇条書きの文頭記号として使用している。（章ごとに記号が混在している）【改行】→【適合例】全ての章において、第1レベルの箇条書き記号を「・」で統一している。 |
| 箇条書きのルールに適合している | 同一の箇条書きリスト内において、末尾の表現形式（体言止め、または句点付きの文章）が整合しており、論理的な粒度が揃っている。 | ★新規追加【改行】【非適合例】・データの入力【改行】・ファイルを削除する。（体言止めと終止形が混在している）【改行】→【適合例】・データの入力【改行】・ファイルの削除 |
| 図表の記載ルールに適合している | 全ての図表に対して、一連の通し番号と内容を簡潔に示す名称（タイトル）が、規定の配置場所（図の下部、表の上部等）に漏れなく付与されている。 | 【非適合例】図.（タイトルが記載されておらず、番号のみとなっている）【改行】→【適合例】図3-1 ネットワーク構成図 |
| 図表の記載ルールに適合している | 図表のタイトルおよび図中の構成要素（ラベル、データ、凡例等）が、本文の説明文や外部設計等の定義内容と完全に一致しており、矛盾が生じていない。 | 【非適合例】図2-1 ログインフローという名称だが、図の中身はログアウト処理の遷移図になっている。（タイトルと内容が一致していない）【改行】→【適合例】図2-1 ログインフローという名称の通り、ログイン処理の画面遷移図が掲載されている。 |
| 各種の記法（状態遷移表，UMLなど）に則している | 状態遷移図や状態遷移表において、遷移を誘発するイベントと実行されるアクションが、定義された記法（区切り文字や記述順序等）に従い、網羅的に記述されている。 | ★新規追加【改行】【非適合例】状態Aから状態Bへ矢印のみが引かれ、遷移条件が記述されていない。（記法に従わず内容が理解しにくい）【改行】→【適合例】「ボタン押下/画面更新」のように、イベントとアクションを規定のセパレータで区切り記述している。 |
| 各種の記法（状態遷移表，UMLなど）に則している | UML（クラス図、シーケンス図等）において、要素間の関係性（継承、実現、依存、関連等）を示す線や矢印の形状が、OMG等の標準記法の定義に基づき厳格に使い分けられている。 | ★新規追加【改行】【非適合例】クラス図において、継承関係と関連の線をすべて同じ実線の矢印で描画している。（UMLの記法に則っていない）【改行】→【適合例】継承には白抜きの三角矢印、関連には通常の実線矢印を用いている。 |
| 用字用語が正しい | 変換ミス、タイピングミスによる誤字、および格助詞の欠落や重複といった脱字を排除し、正確な日本語で記述されている。 | 【非適合例】「作成した機能をデスとして実行する」（「テスト」が「デス」になっている誤字）【改行】→【適合例】「作成した機能をテストとして実行する」 |
| 用字用語が正しい | プロジェクト指定の用語集、または公用文の表記ルール（送り仮名の付け方等）に基づき、表記揺れのない語彙を文書全体で一貫して選択している。 | ★新規追加【改行】【非適合例】同一文書内で「サーバ」と「サーバー」の表記が混在している。（表記揺れが発生している）【改行】→【適合例】プロジェクトの用語定義に従い「サーバ」という表記に統一している。 |
