| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 文書の読み手（属性や特性）を記述している | 文書の冒頭または共通事項において、想定する読者の職種、担う役割、および前提となる技術的知識やスキルレベルを明記している。 | ★新規追加【改行】【非適合例】本書は、システムの仕様をまとめたものである。(非適合の理由：読み手の役割や必要なスキルに関する記述がなく曖昧であるため)【改行】→【適合例】本書の想定読者は、本システムのバックエンドAPIを実装する開発エンジニアである。RESTful API設計の基本概念と、認証基盤の仕組みを理解していることを前提とする。 |
| 文書の読み手の範囲（提出先,配布先,開示範囲）を記述している | 文書の管理情報または前提条件として、文書の提出先組織、配布対象となるチーム、および情報の開示範囲（社内限定、社外秘など）を明確に定義している。 | ★新規追加【改行】【非適合例】本書はプロジェクト計画書であり、フェーズごとの成果物やスケジュールを記載する。(非適合の理由：測定項目の意図から外れ、文書の提出先や情報開示の範囲が記載されていないため)【改行】→【適合例】本書の提出先はシステム企画部であり、配布範囲は開発ベンダーのプロジェクトメンバーのみとする。また、本ドキュメントの取扱いは「社外秘」とする。 |
| 文書の適用範囲を記述している | 文書の対象となるシステムコンポーネント、適用される業務領域、またはプロジェクトのフェーズ（スコープ）の境界線を明記している。 | ★新規追加【改行】【非適合例】システム要件定義書において、顧客のニーズに基づく機能要件を定義する。(非適合の理由：要件定義の内容はあるが、どのシステム範囲や業務領域が対象となるか適用範囲が不明であるため)【改行】→【適合例】本システム要件定義書の適用範囲は、次期ECサイト構築プロジェクトにおける「フェーズ1（商品検索およびカート機能）」のシステムに限定する。決済連携フェーズはスコープ外とする。 |
| 文書の目的を記述している | 文書を作成する意図、および後続プロセスでレビュアーや開発者がどのように活用すべきかという利用目的を記載している。 | ★新規追加【改行】【非適合例】本ソフトウェア設計書には、目的と無関係な機能や仕様が多く記述されている。(非適合の理由：文書を読むことで何を達成すべきかという本来の目的が記載されていないため)【改行】→【適合例】本ソフトウェア設計書の目的は、基本設計書で定義された機能要件を満たすための内部構造を定義し、実装担当者がコーディングを行うための仕様を提示することである。 |
| 文書の目的を達成するために必要な項目を記述している | 要求仕様書の目的を達成するため、前提となる開発の目的および対象となる開発スコープを必須項目として記載している。 | ★新規追加【改行】【非適合例】本要求仕様書では、システムの具体的な機能と画面遷移について説明する。(非適合の理由：要求を定義する上で前提となる開発の目的が明示されていないため)【改行】→【適合例】本要求仕様書では第1章に、業務効率化による残業時間20%削減という「開発の目的」、および顧客管理システムの改修という「開発のスコープ」を記載している。 |
| 文書の目的を達成するために必要な項目を記述している | システムの包括的な要件を定義するため、要求仕様書内に非機能要求（性能、セキュリティ、可用性など）を記載する専用の章を設けている。 | ★新規追加【改行】【非適合例】要求仕様書には、業務フローに基づく機能要件のみが定義されている。(非適合の理由：要件を網羅するために必要な非機能要件の項目が欠落しているため)【改行】→【適合例】要求仕様書には、機能要件とは別に非機能要求を記載する章がある。 |
| 文書の目的を達成するために必要な項目を記述している | 設計作業の意図や方針を明確にするため、設計書内にシステム全体の設計目的やアーキテクチャ選定の背景を記載している。 | ★新規追加【改行】【非適合例】本システムのデータベースには特定のRDBMSを採用し、テーブル定義を進める。(非適合の理由：設計に必要な、アーキテクチャを選定した目的や背景となる項目が記載されていないため)【改行】→【適合例】設計書に設計の目的として「高可用性の実現」を掲げ、これを達成するためにクラスタリング構成を採用する背景を記載している。 |
| 文書の目的を達成するために必要な項目を記述している | 設計の持続性を確保するため、設計書内に将来的な機能追加や改訂開発（拡張性）を想定した設計の意図を記述している。 | ★新規追加【改行】【非適合例】ユーザー認証機能は、現在の仕様に合わせてIDとパスワードのみの単一認証でハードコーディングする。(非適合の理由：将来の改訂開発を想定した設計の目的や方針が示されていないため)【改行】→【適合例】将来の改訂開発（マルチテナント化）を想定した設計目的として、データベースの論理設計においてテナントIDを複合主キーとして持たせる方針を記述している。 |
| 文書の目的を達成するために必要な項目を記述している | 会議の目的である情報共有と合意形成を記録するため、議事録の標準フォーマットとして、開催日、参加者、議題、および議決結果を記載する欄を設けている。 | ★新規追加【改行】【非適合例】前回のミーティングでは、新機能の方向性について話し合われ、A案が良さそうだと意見が一致した。(非適合の理由：議事録として必要な開催日、参加者、正式な議決結果などの必須項目が設けられていないため)【改行】→【適合例】議事録には、開催日、参加者、議題、議決結果を書く欄を設けている。 |
