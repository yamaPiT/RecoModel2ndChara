| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 視点を統一している | 業務フロー図やUI遷移図の解説文において、ユーザを唯一の主語とし、能動態を用いた一貫性のある記述を行っている。 | ★新規追加【改行】【非適合例】ユーザがボタンを押下すると、システムは画面を遷移させ、次にユーザは入力を行う。（同一文内で操作主体と処理主体の視点が混在している）。【改行】→【適合例】ユーザは、スマートウォッチのボタンを押下して次画面へ遷移し、必要な情報を入力する。 |
| 視点を統一している | 機能仕様書や内部設計書において、システムまたは該当コンポーネントを主語とし、処理内容を統一した視点で記述している。 | ★新規追加【改行】【非適合例】心拍数計測機能において、ユーザがセンサーに触れると数値が表示される。（設計書において動作主体がシステムではなくユーザ視点のみで記述されている）。【改行】→【適合例】心拍数計測機能は、センサーからの入力を検知し、算出された心拍数を画面に表示する。 |
| 行為の主体を特定できる | 要件定義やセキュリティ設計において、各操作の実行責任を持つアクター（ロール名）やハードウェア、ソフトウェアの名称を主語に含めている。 | 【非適合例】誰がシステムの操作権限を持っているか、文書内で特定できない。（操作権限を持つ主体が不明確である）。【改行】→【適合例】要件仕様書において、管理責任者がユーザのアカウント削除権限を持つことを明記している。 |
| 箇条書きまたは図表を適切に使用している | 文書内に挿入されたすべての図および表に対して、一意に識別可能な通し番号（図表番号）と、内容を簡潔に示す題名（キャプション）を付与している。 | 【非適合例】図表にキャプションが付いておらず、本文で参照されていない。（図表の識別情報が欠落している）。【改行】→【適合例】図3-1 スマートウォッチ外観図のように、図表に項番およびキャプションを付けている。 |
| 箇条書きまたは図表を適切に使用している | 本文から図表を参照する際、「下図」「上表」といった曖昧な表現を避け、図表番号を用いて指示対象を厳密に特定している。 | ★新規追加【改行】【非適合例】下図の通り、データの流れを定義する。（複数の図がある場合にどの図を指しているか特定できない）。【改行】→【適合例】図4-2 データフロー図に示す通り、各処理間のデータ受け渡しを定義する。 |
| インデントまたは改行を効果的に使用している | 文書の論理構造に基づき、見出しレベルに応じた字下げ（インデント）や、視覚的な段落の区切りとなる行間設定を、全編を通じて統一した規約で適用している。 | 【非適合例】見出しと本文でインデントや改行が不統一で、読みづらい。（視覚的な構造が整理されていない）。【改行】→【適合例】章、節の見出しレベルに応じた字下げを行い、本文との境界に適切な空白行を挿入している。 |
| 読みやすいレイアウトやフォント種を使用している | 印刷時や部分的な参照時の取り違えを防止するため、全ページのヘッダーまたはフッターの定位置に、管理対象としての正式な文書名を一貫して表示している。 | 【非適合例】ヘッダーに文書名が記載されていない。（印刷時などにどの文書の一部か判別できない）。【改行】→【適合例】全ページのヘッダーに、文書名「基本設計書」を記載している。 |
| 読みやすいレイアウトやフォント種を使用している | 文書の全ページに対して、総ページ数および当該ページ番号（例：1/20）を、ページ内の統一された位置にノンブルとして付与している。 | 【非適合例】ページ番号がない。（会議等で特定の箇所を指し示すことが困難である）。【改行】→【適合例】本文の各ページ下部中央に、ノンブル（ページ番号）を記載している。 |
