| 文体を統一している | プロジェクトの文書執筆規約に基づき、文書全体の文末表現を常体（である調）または敬体（です・ます調）のいずれかに一貫して統一している。 | ★新規追加【改行】【非適合例】本システムは、ユーザー情報を管理する。また、パスワードの有効期限は90日としています。（非適合の理由：「です・ます調」と「である調」が混在している）【改行】→【適合例】本システムは、ユーザー情報を管理する。また、パスワードの有効期限は90日としている。 |
| 用語および表記法を統一している | プロジェクトの用語定義集や表記ガイドラインに準拠し、同一文書内で同義語の表記揺れ（かな、漢字、アルファベット、送り仮名など）を発生させず同一の表記を採用している。 | ★新規追加【改行】【非適合例】データベースサーバの構築後、Webサーバーとの通信確認を行う。（非適合の理由：同一文書内で異なる表記や用語が使用されている）【改行】→【適合例】データベースサーバの構築後、Webサーバとの通信確認を行う。 |
| 用語を使い分けている | 文書の想定読者（顧客、外部ベンダー、運用担当者など）の理解度を考慮し、組織内でのみ通用する隠語や特定ドメインの専門用語を排除して汎用的な用語に置換している。 | ★新規追加【改行】【非適合例】夜間バッチがコケた場合、シス管にアラートを飛ばす。（非適合の理由：他社が読み手になる文書において、仲間うちだけでしか通じない用語を使用している）【改行】→【適合例】夜間バッチ処理が異常終了した場合、システム管理者にアラート通知を行う。 |
| 用語（専門用語，略語など）および記号を定義している（「再利用性」に関連） | 文書内で使用する専門用語や略語について、初出時にフルスペルや意味を明記するか、プロジェクト共通の用語定義集に登録して解釈のブレを防止している。 | ★新規追加【ク改行】【非適合例】本機能は、BFF層でデータの集約処理を行う。（非適合の理由：略語が事前定義なしに使用されている）【改行】→【適合例】本機能は、BFF（Backends For Frontends）層でデータの集約処理を行う。 |
| 用語（専門用語，略語など）および記号を定義している（「再利用性」に関連） | 閾値や制約事項を示す条件式について、使用する記号（不等号や波線など）が示す数値の境界値（以上、以下、未満など）を解釈の余地なく厳密に言語化している。 | ★新規追加【改行】【非適合例】パスワードの文字数は8文字〜16文字とする。（非適合の理由：値や範囲が曖昧で、正確に定義されていない）【改行】→【適合例】パスワードの文字数は8文字以上16文字以下とする。 |
| 用字用語が適切である | 日本語で表現可能な一般的な動作や状態について、読解を妨げるIT業界特有の不必要なカタカナ語を使用せず、標準的な和語や漢語を用いて記述している。 | ★新規追加【改行】【非適合例】要件定義でアグリーした仕様をベースに設計をフィックスする。（非適合の理由：不必要なカタカナ表現が使用され、文が読みにくくなっている）【改行】→【適合例】要件定義で合意した仕様に基づき設計を確定する。 |
| 用字用語が適切である | 接続詞や副詞などの表記において、JTF日本語標準スタイルガイドや公用文作成の要領などで漢字よりひらがなが推奨されているルールに準拠している。 | ★新規追加【改行】【非適合例】サーバを停止する際は、予めユーザーへ告知する。（非適合の理由：漢字よりひらがなが推奨される表記に従っていない）【改行】→【適合例】サーバを停止する際は、あらかじめユーザーへ告知する。 |
| 用字用語が適切である | 一般的な用字用語集（記者ハンドブック等）やプロジェクト固有の執筆ルールに基づき、同音異義語や類義語の中から文脈の意図を正確に伝達する用語を選択している。 | ★新規追加【改行】【非適合例】システム障害によって発生した顧客の損害を保障する。（非適合の理由：意味合いとして不適切な同音異義語が使用されている）【改行】→【適合例】システム障害によって発生した顧客の損害を補償する。 |
| 記号類・括弧類を適切に使用している | プロジェクトのドキュメント規約に基づき、引用符（ストレートクォートとカーリークォート等）や各種括弧類を、画面名、ボタン名、コードの文字列リテラルなどの対象要素ごとに厳密に使い分けている。 | ★新規追加【改行】【非適合例】設定ファイルのキー名として “timeout" を指定する。（非適合の理由：異なる種類の引用符を同じ意味で混同して使用している）【改行】→【適合例】設定ファイルのキー名として "timeout" を指定する。 |
