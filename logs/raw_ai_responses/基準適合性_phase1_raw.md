| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 職務分掌に則った権限委譲などのビジネスルールに従っている | 職務分掌規程等の組織定義に基づき、各役割に対して適切な操作権限を割り当て、不正なアクセスや操作を排除する制御を記述している。 | 【非適合例】アクセス権限が設定されていないため、所属部署に関係なく全ユーザーが発注処理を無制限に実行できる。(権限分離の欠如)【改行】→【適合例】職務分掌規程に基づき、所属部署以外から発注処理を行えないよう、ユーザーの役割に応じたアクセス権限を制限する。 |
| 国内だけでなく、グローバルな環境にも適合している | 展開対象となる各国の排出ガス規制や安全基準等の法的要件を確認し、仕向地ごとの基準値をすべて充足する設計値を記述している。 | ★新規追加【改行】【非適合例】エンジン制御のパラメータを国内基準のみに合わせて設計しているため、北米の排出ガス規制値を満たさない恐れがある。(特定地域のみの考慮)【改行】→【適合例】エンジン制御システムにおいて、排出ガスが日本および北米等の各仕向地の規制値をすべて満たすよう、制御パラメータを設計する。 |
| 国内だけでなく、グローバルな環境にも適合している | サービス提供範囲が多国籍にわたる場合、特定国のみで有効なインフラや番号体系に依存せず、すべての利用環境からアクセス可能な代替手段を記述している。 | 【非適合例】問い合わせ先として国内専用の無料通話番号のみを記載しており、国外からは電話をかけられない。(グローバル対応の欠如)【改行】→【適合例】問い合わせ手段として、国外からも接続可能な電話番号（国番号付き）およびメールアドレスを併記する。 |
| 技術領域の基準または標準に準拠している | 採用する技術要素、アルゴリズム、UI設計等が、既存の特許権や著作権を侵害していないことを特許調査等に基づき確認し、その結果を記述している。 | 【非適合例】競合他社が保有する特許技術と類似の検索アルゴリズムを採用しているが、侵害のリスク調査がなされていない。(特許侵害の懸念)【改行】→【適合例】採用する検索アルゴリズムについて特許調査を実施し、他社の特許権を侵害していないことを確認した旨を明記する。 |
| 技術領域の基準または標準に準拠している | 個人情報保護法や業界特有のセキュリティ標準に基づき、個人情報データベースの暗号化、アクセスログ保存、匿名化等の具体的な管理手法を記述している。 | ★新規追加【改行】【非適合例】個人情報データベースの管理方法について、具体的な保護ルールが明文化されていない。(基準準拠の欠如)【改行】→【適合例】個人情報保護法および社内規定に基づき、個人情報データベースには特定の管理者のみがアクセスできるよう制限し、操作ログを保存する。 |
| 文書標準に準拠している | プロジェクトで定義された文書構成、見出しレベル、フォント、および指定のテンプレート項目をすべて網羅して記述している。 | 【非適合例】プロジェクト指定の基本設計書テンプレートを使用せず、独自の章立てや様式で記述されている。(標準フォーマットの無視)【改行】→【適合例】プロジェクト標準の文書テンプレートに従い、各章節の構成および記述ルールを遵守して作成する。 |
| 文書標準に準拠している | 他の仕様書、法令、外部規格等から情報を引用または参照する場合、その情報源の名称、バージョン、発行元を適切に記述している。 | ★新規追加【改行】【非適合例】外部システムとの通信仕様について、相手方の資料を一部引用しているが、どの資料に基づいているかが不明である。(引用元の不明確)【改行】→【適合例】「外部接続仕様書（第2.1版）」の通信シーケンスに基づき、本システムのインタフェース制御を記述することを明記する。 |
| 事実と意見を書き分けている | 技術的制約、テスト結果、顧客提示の絶対条件等の客観的な事実は、推測や主観的な解釈を交えず、証跡に基づいた断定表現で記述している。 | ★新規追加【改行】【非適合例】テスト結果について、「概ね問題ないと思われる」といった主観的な表現で記述されている。(事実の主観化)【改行】→【適合例】テスト結果について、「全100ケースのテストを完了し、不具合発生数は0件である」という事実を客観的に記述する。 |
| 事実と意見を書き分けている | 要求仕様案、設計判断、デバッグ方針等の主観的な意見や判断は、事実と混同しないよう、文末表現や項目を分けることで、作成者の解釈であることを明示している。 | ★新規追加【改行】【非適合例】設計結果の記述において、なぜそのアルゴリズムを採用したのかという設計者の判断が、システムの仕様そのもののように記述されている。(判断根拠の混同)【改行】→【適合例】設計結果に対し、「計算速度の向上を図るため、XXアルゴリズムの採用が最適であると判断した」という設計者の意見（判断根拠）を、仕様記述とは明確に分けて記述する。 |
