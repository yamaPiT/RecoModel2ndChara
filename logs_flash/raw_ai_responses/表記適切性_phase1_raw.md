| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| :--- | :--- | :--- |
| 文体を統一している | 文書全体または章などの論理単位において、敬体（です・ます調）または常体（である調）のいずれか一方を選択し、混在させずに記述している。 | 【非適合例】本機能は情報の検索を行います。また、結果を一覧で表示する。（「です・ます調」と「である調」が混在している）【改行】→【適合例】本機能は情報の検索を行います。また、結果を一覧で表示します。 |
| 用語および表記法を統一している | 同一の対象、概念、または動作を示す言葉について、全編を通じて表記（漢字・ひらがな・カタカナ・英数字）を統一して記述している。 | 【非適合例】画面上のユーザをクリックすると、ユーザー詳細が表示される。（「ユーザ」と「ユーザー」の表記が同一文書内で混在している）【改行】→【適合例】画面上のユーザーをクリックすると、ユーザー詳細が表示される。 |
| 用語および表記法を統一している | 同一の対象や処理を指す語彙（用語）が揺れないよう、用語定義集や標準的な技術用語に基づき、名称を一貫させて記述している。 | ★新規追加【改行】【非適合例】データをサーバに保存した後、当該レコードをコミットする。（「保存」と「コミット」が同じ動作を指して混在している）【改行】→【適合例】データをサーバに保存した後、当該レコードを保存する。 |
| 用語を使い分けている | 読者の立場や専門性を考慮し、社内特有の隠語（ジャーゴン）を避け、対象読者が共通して理解できる一般的な用語を選択して記述している。 | ★新規追加【改行】【非適合例】本件のASISにおけるボトルネックは、ホスト側のバッチ処理にある。（社外の読み手に対して「ASIS」などの専門用語を説明なしに使用している）【改行】→【適合例】本システムの現状（As-Is）における問題点は、基幹システムのバッチ処理にある。 |
| 用語（専門用語，略語など）および記号を定義している（「再利用性」に関連） | 独自の意味を持つ専門用語や略語について、初出箇所または用語定義セクションにおいて、正式名称や定義内容を明記している。 | ★新規追加【改行】【非適合例】APIを利用してデータを取得する。（「API」という略語の定義や正式名称が文書内に明記されていない）【改行】→【適合例】Application Programming Interface（以下、API）を利用してデータを取得する。 |
| 用語（専門用語，略語など）および記号を定義している（「再利用性」に関連） | 判定条件や仕様に含まれる値の範囲について、境界値（以上・以下・未満）や単位、および特殊な演算記号の意味を厳密に定義している。 | 【非適合例】大容量のファイルを送信する場合は注意が必要である。（「大容量」が具体的に何バイト以上を指すか定義されていない）【改行】→【適合例】100MB以上のファイル（以下、大容量ファイル）を送信する場合は、事前に管理者へ通知する。 |
| 用字用語が適切である | 一般的な日本語で表現可能な語彙について、過度なカタカナ語や外来語を避け、文脈に応じた平易な日本語を選択して記述している。 | 【非適合例】本プロジェクトのスキームをアグリーした後にアサインを行う。（不必要なカタカナ表現が使用され、文が読みにくくなっている）【改行】→【適合例】本プロジェクトの計画に合意した後に、担当者の割り当てを行う。 |
| 用字用語が適切である | 補助動詞、形式名詞、接続詞など、公用文や一般的な執筆ガイドラインにおいて「ひらがな表記」が推奨されている語彙を適切に使い分けて記述している。 | ★新規追加【改行】【非適合例】データの入力、及び、保存を併せて行う。（「及び」「併せて」など、ひらがな表記が推奨される接続詞等を漢字で記述している）【改行】→【適合例】データの入力、および、保存をあわせて行う。 |
| 用字用語が適切である | 組織やプロジェクトで採用している標準的な用字用語集（記者ハンドブック等）や独自の規程に従い、一貫した表記ルールで記述している。 | ★新規追加【改行】【非適合例】日付は 2024/03/25 と記載する。（プロジェクト規定で「YYYY年MM月DD日」形式と定められているが、異なる形式で記述している）【改行】→【適合例】日付は 2024年03月25日 と記載する。 |
| 用字用語が適切である | 複数の意味を持つ曖昧な語を排除し、その文脈において最も正確な動作や状態を示す具体的な技術用語を選択して記述している。 | ★新規追加【改行】【非適合例】保存期間を過ぎた古いデータを消す。（「消す」という表現が、物理削除なのか論理削除なのか不明確である）【改行】→【適合例】保存期間を過ぎたデータを物理削除する。 |
| 記号類・括弧類を適切に使用している | 引用、強調、変数、入力値など、目的ごとに使用する記号（「」、（）、『』、""）の役割を定義し、そのルールに従って厳格に使い分けて記述している。 | 【非適合例】ユーザー名は「Admin」とし、パスワードは（password）とする。（同じ入力値の明示に対して異なる括弧を同じ意味で使用している）【改行】→【適合例】ユーザー名は「Admin」とし、パスワードは「password」とする。 |
| 記号類・括弧類を適切に使用している | 類似した形状を持つ記号（プライム、クォート各種など）について、文字コードや用途の違いを認識し、技術的に正しい記号を選択して記述している。 | ★新規追加【改行】【非適合例】SQL文の条件に 'Admin' を指定する。（ストレートクォートを使用すべき箇所で、形状の似た別の記号を使用している）【改行】→【適合例】SQL文の条件に 'Admin' を指定する。 |
