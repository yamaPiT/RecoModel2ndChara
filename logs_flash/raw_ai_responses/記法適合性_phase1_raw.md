| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 箇条書きのルールに適合している | 箇条書きの文頭記号（中黒、番号、チェックなどの記号）が、プロジェクトの規定または文書全体で統一されている。 | ★新規追加【改行】【非適合例】第1章では「・」を使用し、第2章では「-」を箇条書きの文頭記号として使用している。(章ごとに記号が混在している）【改行】→【適合例】全ての章において、第1レベルの箇条書き記号を「・」で統一している。 |
| 箇条書きのルールに適合している | 箇条書きの各項目の文末表現（体言止め、または句点付きの文章）が、同一のリスト内で統一されている。 | ★新規追加【改行】【非適合例】・データの入力【改行】・ファイルを削除する。(体言止めと終止形が混在している）【改行】→【適合例】・データの入力【改行】・ファイルの削除 |
| 図表の記載ルールに適合している | 全ての図表に対して、一連の通し番号と内容を簡潔に示す名称（タイトル）を付与している。 | 【非適合例】図.（タイトルが記載されておらず、番号のみとなっている）【改行】→【適合例】図 3-1 ネットワーク構成図 |
| 図表の記載ルールに適合している | 図表のタイトルおよび図中の構成要素が、本文での説明内容や定義された仕様と矛盾なく対応している。 | 【非適合例】図 2-1 ログインフローという名称だが、図の中身はログアウト処理の遷移図になっている。(タイトルと内容が一致していない）【改行】→【適合例】図 2-1 ログインフローという名称の通り、ログイン処理の画面遷移図が掲載されている。 |
| 各種の記法（状態遷移表，UMLなど）に則している | 状態遷移図や状態遷移表において、遷移の契機となるイベントと遷移後のアクションが規定の書式に基づき漏れなく記述されている。 | ★新規追加【改行】【非適合例】状態Aから状態Bへ矢印のみが引かれ、遷移条件が記述されていない。(記法に従わず内容が理解しにくい）【改行】→【適合例】「ボタン押下 / 画面更新」のように、イベントとアクションを規定のセパレータで区切り記述している。 |
| 各種の記法（状態遷移表，UMLなど）に則している | UML（クラス図、シーケンス図等）において、要素間の関係性を示す線や矢印の形状が、標準記法の定義に基づき正しく使い分けられている。 | ★新規追加【改行】【非適合例】クラス図において、継承関係と関連の線をすべて同じ実線の矢印で描画している。(UMLの記法に則っていない）【改行】→【適合例】継承には白抜きの三角矢印、関連には通常の実線矢印を用いている。 |
| 用字用語が正しい | 変換ミス、タイピングミスによる明らかな誤字、および助詞の欠落などの脱字を排除し、正確な日本語で記述している。 | 【非適合例】「作成した機能をデスとして実行する」(「テスト」が「デス」になっている誤字）【改行】→【適合例】「作成した機能をテストとして実行する」 |
| 用字用語が正しい | プロジェクトで指定された用語集や公的な表記ルール（送り仮名の付け方等）に基づき、表記揺れのない語彙を選択している。 | ★新規追加【改行】【非適合例】同一文書内で「サーバ」と「サーバー」の表記が混在している。(表記揺れが発生している）【改行】→【適合例】プロジェクトの用語定義に従い「サーバ」という表記に統一している。 |
