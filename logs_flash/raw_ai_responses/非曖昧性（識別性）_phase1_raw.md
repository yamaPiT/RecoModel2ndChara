| 測定項目 | 測定基準 | 記述例（適合例／非適合例） |
| ------ | ------ | ------ |
| 動作を特定できる | 動作の具体的な内容や代入する値を、システム設計方針で定義された特定の数値や状態名と照らし合わせて客観的に判定できる。 | 【非適合例】変数Aをリセットする。 (リセットの動作が不明)【改行】→【適合例】変数Aに初期値の1を代入する。 |
| 状態を特定できる | 状態遷移や動作のトリガーとなる条件において、しきい値や継続回数などの定量的な数値を定義し、判定の境界を明示している。 | ★新規追加【改行】【非適合例】通信が不安定な場合に異常状態へ遷移する。 (不安定の基準が不明)【改行】→【適合例】センサAの読み取り値が、2回連続して99を超えた場合に過負荷状態に遷移する。 |
| 状態を特定できる | 性質や状況を表す形容詞を用いる際、用語集で定義された基準値や客観的に測定可能な単位を併記し、解釈のブレを排除している。 | ★新規追加【改行】【非適合例】大量のデータを高速に処理する。 (「大量」「高速」の定義がない)【改行】→【適合例】1秒間に10,000件のデータを、300ミリ秒以内で処理する。 |
| 対象を特定できる | 動作や説明の対象となるハードウェア、ソフトウェアコンポーネント、またはモジュールの名称を、構成図や用語定義と一致する正式名称で記述している。 | 【非適合例】構造を説明する。 (何の構造か不明)【改行】→【適合例】次の章で、流量センサのセンシング構造を説明する。 |
| 対象を特定できる | 物理的な配置や画面上の位置を示す際に、相対的な表現を避け、図表番号や座標、または識別IDを用いて一意に特定している。 | 【非適合例】スイッチをパネルの上部に取り付ける。 (「上部」だけでは位置を特定できない)【改行】→【適合例】スイッチを図2の位置に取り付ける。 |
| 対象を特定できる | 動詞が作用する対象（目的語）を必ず含め、その動作がどのデータやリソースに対して実行されるかを明記している。 | ★新規追加【改行】【非適合例】対象物を明示せずに、「分割する」や「制御する」とだけ書く。 (動作の対象となる必須補語が欠落している)【改行】→【適合例】受信した通信パケットを、ヘッダ部とペイロード部に分割する。 |
| 違う内容のものを同じ名前で定義しない | 同一の名称や演算子を異なる文脈で使用する場合、適用される型や条件ごとの動作の違いを、外部仕様やインターフェース定義に基づいて明確に区別している。 | ★新規追加【改行】【非適合例】多重定義が文書内で不適切に使用されている。 (文脈による定義の差異が不明確)【改行】→【適合例】'+'演算子：【改行】整数では算術和と定義すると、1+2は3となる【改行】文字列では連結として定義すると、"ab"+"cd"は"abcd"となる。 |
| 意味を特定しやすい助詞を使用している | 起点や開始を表す際には、比較の意味を持つ「より」を避け、起点として一義的に解釈される「から」を使用している。 | 【非適合例】このテスト項目より、テストを始める。 (起点には「より」ではなく「から」を推奨する)【改行】→【適合例】このテスト項目から、テストを始める。 |
| 意味を特定しやすい助詞を使用している | 動作の終点や帰着点を示す際には、方向を表す「へ」ではなく、到達対象を特定する「に」を使用している。 | 【非適合例】設計のレビュー対応が完了したら、実装工程へ進む。 (終点には「へ」ではなく「に」を推奨する)【改行】→【適合例】設計のレビュー対応が完了したら、実装工程に進む。 |
